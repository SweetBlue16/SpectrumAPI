======================================================================
                 GUÍA RÁPIDA: CONVENTIONAL COMMITS
======================================================================

Esta es la convención estándar para redactar el historial del proyecto. 
Cada commit debe contar una historia clara de qué cambió y por qué.

----------------------------------------------------------------------
1. ESTRUCTURA BÁSICA
----------------------------------------------------------------------
<tipo>(<ámbito opcional>): <descripción corta>
<LÍNEA EN BLANCO>
[cuerpo detallado opcional]
<LÍNEA EN BLANCO>
[pie de página opcional]

----------------------------------------------------------------------
2. TIPOS DE COMMIT (El "Qué")
----------------------------------------------------------------------
Usa estos prefijos obligatorios para categorizar tu trabajo:

* feat     : Una nueva característica o funcionalidad.
             Ej: feat(api): añade endpoint para consultar usuarios
* fix      : Corrección de un error o bug.
             Ej: fix(auth): resuelve excepción de WCF en el login
* test     : Añadir o corregir pruebas unitarias/integración.
             Ej: test(core): añade pruebas unitarias para el cálculo de viabilidad AHP
* refactor : Cambios en el código que no añaden funciones ni corrigen bugs, 
             sino que mejoran la estructura o arquitectura.
             Ej: refactor(db): separa Entity Framework Core de la lógica de negocio
* docs     : Cambios exclusivos en la documentación (README, Swagger, etc).
             Ej: docs: actualiza instrucciones de despliegue en README
* style    : Cambios visuales en el código (formateo, espacios, comillas).
             Ej: style: formatea archivos de la capa de acceso a datos
* perf     : Mejoras de rendimiento en el código existente.
             Ej: perf(ui): optimiza el renderizado de la cuadrícula en WPF
* chore    : Tareas de mantenimiento, actualización de dependencias, 
             configuración de herramientas.
             Ej: chore: actualiza paquetes de .NET 8 a la última versión
* ci       : Cambios en los archivos de integración continua (GitHub Actions).
             Ej: ci: añade validación de commits con husky

----------------------------------------------------------------------
3. ÁMBITO / SCOPE (Opcional, pero recomendado)
----------------------------------------------------------------------
Va entre paréntesis y en minúsculas. Indica la capa, módulo o 
componente afectado para dar contexto rápido.

Ejemplos comunes: (api), (ui), (db), (auth), (router), (core)

----------------------------------------------------------------------
4. REGLAS PARA LA DESCRIPCIÓN CORTA (El "Resumen")
----------------------------------------------------------------------
- MÁXIMO 50 caracteres.
- Usa MODO IMPERATIVO (como si dieras una orden).
  BIEN: "añade", "corrige", "elimina", "actualiza".
  MAL:  "añadido", "corrigiendo", "se eliminó".
- Todo en minúsculas (preferentemente).
- SIN punto final.

REGLA DE ORO: Si aplicas este commit, el código va a: <descripción>
Ej: [Si aplicas este commit, el código va a] "añadir validación de JWT"

----------------------------------------------------------------------
5. EL CUERPO Y EL PIE DE PÁGINA (Para commits complejos)
----------------------------------------------------------------------
Si el cambio es grande y el título no basta, deja una línea en blanco
y explica el POR QUÉ y el QUÉ (el cómo se ve en el código).

EL CUERPO:
Explica el problema original y la solución adoptada. Trata de que 
las líneas no pasen de 72 caracteres de largo para que sea legible 
en cualquier terminal.

EL PIE DE PÁGINA:
- Úsalo para indicar si hay "BREAKING CHANGES" (cambios que rompen la 
  compatibilidad con versiones anteriores).
- Úsalo para cerrar tickets automáticamente en GitHub/Jira.

----------------------------------------------------------------------
EJEMPLO DE UN COMMIT COMPLETO Y PERFECTO:
----------------------------------------------------------------------
feat(auth): implementa sistema de roles con JWT

Anteriormente los usuarios tenían acceso global a todos los endpoints 
de la API, lo que representaba una vulnerabilidad de seguridad.

Se integró la generación y validación de tokens JWT en el middleware.
Ahora cada petición requiere un token válido en el header Authorization.

BREAKING CHANGE: Los clientes del frontend ahora deben enviar el header
Authorization con el prefijo Bearer.

Closes #45
